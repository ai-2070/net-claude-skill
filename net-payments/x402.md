# x402 — the payment wire (byte-preserved, quarantined)

Everything that parses or interprets x402 lives in `payments/src/x402/`. **All
x402 spec churn is quarantined here** — no x402 type leaks into Net core, Net
envelopes, or per-network branches. Pinned spec revision:
`specs/x402-specification-v2.md` at x402-foundation/x402 commit
`087922a5eecc06ea773636b75df205814ba295b5` (2026-05-29). This build speaks
`X402_VERSION = 2`.

Read `concepts.md` for *why* byte-preservation matters; this file is the
*how* and the exact structures.

## `X402Carry<T>` — the load-bearing type (`x402/mod.rs`)

Holds the **original bytes** of an x402 JSON document alongside a validated,
read-only parsed view. Net envelopes embed the carry; **the carry serializes
as base64 of the original bytes** — not nested JSON — so no serializer in any
language can accidentally re-encode the x402 document and break a signature.

```rust
pub struct X402Carry<T: X402View> { /* bytes: Vec<u8>, view: T */ }

X402Carry::from_bytes(bytes: Vec<u8>) -> Result<Self, X402Error>   // parse + validate + PRESERVE (received docs)
X402Carry::author(view: &T) -> Result<Self, X402Error>             // the ONE sanctioned serialization point (locally-originated docs)
carry.bytes() -> &[u8]         // the preserved originals — what signatures cover
carry.view()  -> &T            // validated read-only view (interpretation only)
carry.as_json_str() -> &str
carry.content_hash() -> String // blake3 hex of the preserved bytes (replay-index keys, terms_hash inputs)
```

Invariants (from the source, verbatim intent):
- `bytes` are the exact bytes the document arrived with (whitespace, key
  order, unknown fields, all) — they are what signatures cover and what
  travels.
- The `view` was parsed from those bytes and passed `X402View::validate`.
- **No API re-serializes the view. There is deliberately no
  `to_bytes(&self.view)`.**

`author` is the *only* place serialization through Net types is allowed —
because the document *originates* there, so those bytes become the preserved
originals. **The forbidden move is re-serializing a *received* document.** If
you're building something to sign from a received `PaymentRequirements`, carry
its bytes; never `serde_json::to_vec` it.

`X402View` is the trait every x402 structure implements:

```rust
pub trait X402View: DeserializeOwned {
    const KIND: &'static str;                       // e.g. "PaymentRequirements"
    fn validate(&self) -> Result<(), X402Error>;    // version, CAIP parse, atomic-amount grammar, non-empty fields
}
```

`X402Error`: `NotJson(String)`, `Invalid(String)`,
`UnsupportedX402Version {got, expected}`, `NotBase64(String)`. Views tolerate
unknown fields (the spec is additive; bytes are preserved regardless of what
the view understands).

## The x402 v2 structures (parsed views)

All use `#[serde(rename_all = "camelCase")]` — the on-wire spelling is
camelCase (`payTo`, `maxTimeoutSeconds`, `x402Version`), the Rust field is
snake_case.

### `PaymentRequirements` (`x402/requirements.rs`) — one `accepts[]` entry

```rust
pub struct PaymentRequirements {
    pub scheme: String,             // "exact" (real), "mock" (mock network)
    pub network: String,            // CAIP-2, e.g. "eip155:84532"
    pub amount: String,             // atomic units as a string
    pub asset: String,              // scheme-scoped locator: token contract addr, or ISO-4217 code for fiat rails
    pub pay_to: String,             // recipient wallet address or role constant  (wire: payTo)
    pub max_timeout_seconds: u64,   // ADVISORY on the mesh — the quote envelope's expiry governs  (wire: maxTimeoutSeconds)
    pub extra: Option<Value>,       // scheme-specific (e.g. EIP-712 domain {name, version})
}
requirements.chain() -> Result<ChainId, X402Error>
```

`validate` rejects empty scheme/asset/payTo, zero timeout, bad CAIP network,
and non-atomic amount. Net's registry maps `(network, asset)` to a CAIP-19 id
and policy (see `networks.md`).

### `PaymentPayload` (`x402/payload.rs`) — the client-signed authorization

```rust
pub struct PaymentPayload {
    pub x402_version: u64,          // must be 2
    pub resource: Option<Value>,    // optional echo of the resource being paid for
    pub accepted: PaymentRequirements, // the entry the client accepted, echoed back
    pub payload: Value,             // scheme-specific auth (opaque to Net; the scheme binds it to `accepted`)
    pub extensions: Option<Value>,  // x402 extensions — consumed for interop ONLY, never a substitute for Net identity/consent/billing
}
```

**There is no separate Net intent object** — this payload travels in the
invocation envelope, byte-preserved. Binding of payload→requirements is
x402-internal (scheme-level), and that's the point: Net's quote binds to the
requirements; the *scheme* binds the payment to them.

### `SettlementResponse` + `VerifyResponse` (`x402/settlement.rs`)

Facilitator responses. `POST /settle` → `SettlementResponse`; `POST /verify` →
`VerifyResponse`.

```rust
pub struct SettlementResponse {
    pub success: bool,
    pub error_reason: Option<String>,
    pub payer: Option<String>,
    pub transaction: String,        // tx hash/id on the settlement network
    pub network: String,            // CAIP-2
    pub amount: Option<String>,     // amount DELIVERED — verification checks delivered, never sent (fees)
    pub extensions: Option<Value>,
}
pub struct VerifyResponse { pub is_valid: bool, pub invalid_reason: Option<String>, pub payer: Option<String>, pub extra: Option<Value> }
```

A successful settle must carry a non-empty `transaction`; an invalid verify
must carry an `invalid_reason`.

### `PaymentRequired` (`x402/payment_required.rs`) — the 402 demand

```rust
pub struct PaymentRequired { pub x402_version: u64, pub error: Option<String>, pub resource: Option<Value>, pub accepts: Vec<PaymentRequirements>, pub extensions: Option<Value> }
```

Used by the outbound HTTP 402 client (`http402.md`) to parse an external
API's demand.

## CAIP identifiers (`x402/caip.rs`)

Comparison is **exact and case-sensitive**, per CAIP. Two spellings of the
same on-chain asset (e.g. EIP-55 checksum variants) are *different ids* here;
normalization, if any, is a **registry-policy** decision — never done in the
id type. The cross-language confusion vectors pin this: a verifier that
case-normalizes or trims fails them.

```rust
pub struct ChainId { /* namespace:reference */ }   // e.g. eip155:8453, solana:5eykt…, mock:net
ChainId::parse(&str) -> Result<Self, CaipError>
chain.namespace() / chain.reference() / chain.as_str()
// namespace [-a-z0-9]{3,8}, reference [-_a-zA-Z0-9]{1,32}

pub struct AssetId { /* chain_id/asset_namespace:asset_reference[/token_id] */ }
AssetId::parse(&str) -> Result<Self, CaipError>
asset.chain() / asset.namespace() / asset.reference() / asset.token_id() / asset.as_str()
// asset_namespace [-a-z0-9]{3,8}, asset_reference [-.%a-zA-Z0-9]{1,128}, token_id [-.%a-zA-Z0-9]{1,78}
```

Examples: `eip155:8453/erc20:0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913`
(USDC on Base); `mock:net/token:musd` (the P0 mock asset). `CaipError` is
`Chain(id, reason)` / `Asset(id, reason)`.

Note: the x402 `asset` field is the *scheme-scoped locator* (a bare contract
address like `0x8335…`), not a full CAIP-19 id. The registry is what maps
`(network, asset)` → the CAIP-19 `AssetId` + decimals + policy.

## The `exact` EVM scheme (`x402/schemes/exact_evm.rs`)

The first real scheme: EIP-3009 `transferWithAuthorization`, signed as EIP-712
typed data. This is the authoring side; the signing itself goes through the
`SchemeSigner` seam (see `signer.md`).

```rust
pub struct ExactEvmAuthorization { pub from, pub to, pub value: String, pub valid_after: u64, pub valid_before: u64, pub nonce: String }

exact_evm::chain_id(network: &str) -> Result<u64, X402Error>          // "eip155:8453" -> 8453
exact_evm::typed_data(requirements, auth) -> Result<Value, X402Error> // the eth_signTypedData_v4 TransferWithAuthorization doc
exact_evm::payload_object(auth, signature: &str) -> Value             // the x402 {signature, authorization} payload
```

The EIP-712 domain comes from `requirements.extra {name, version}` + chain id
+ asset contract — spec-carried, cross-checked against the registry. The
validity window derives from the quote's authoritative expiry; the nonce is
quote-derived (32 bytes) so **same-quote retries re-present the identical
authorization** — idempotent at the provider *and* at the token contract's
replay guard. See `signer.md` for the full authoring→signing→payload path.

## The `exact` SVM scheme (`x402/schemes/exact_svm.rs`)

Solana's `exact` scheme is **intent-in / blob-out**: Net builds the typed
`SvmTransferIntent`; the wallet builds and partially signs an SPL
`TransferChecked` versioned transaction. The spec-pinned payload shape is
`payload = {"transaction": "<base64 partially-signed versioned tx>"}` — the
facilitator named by `requirements.extra.feePayer` sponsors fees and completes
the signature set.

```rust
pub struct SvmTransferIntent { pub network, pub mint, pub pay_to, pub amount, pub fee_payer, pub memo: Option<String> }

exact_svm::transfer_intent(requirements) -> Result<SvmTransferIntent, X402Error>  // DERIVED from requirements, never caller-supplied
exact_svm::payload_object(transaction_b64) -> Result<Value, X402Error>            // {"transaction":"<base64>"}, rejects empty/non-base64
```

Intent fields are **derived from the quoted requirements** (`asset` → mint,
`payTo` → recipient, `amount`, `extra.feePayer`, `extra.memo`), never
caller-supplied — so the mismatch class the EVM path has to cross-check away
cannot arise by construction. `transfer_intent` hard-rejects a non-`exact`
scheme, a non-`solana` namespace, a missing/empty `extra.feePayer` (spec-
required), a memo over 256 bytes, or a non-string memo.

Two properties that differ from EVM (see `signer.md` for detail): **Net cannot
decode the returned blob** to re-verify it (that would put SVM transaction
machinery in the money path — the trust chain is the user's wallet + the
facilitator/chain instead), and **same-quote retries may re-author against a
fresh blockhash** → different payload bytes, so idempotency holds at the
**quote**, not at payload byte-identity.

## Amounts are strings, always

x402 amounts are atomic-unit strings; Net parses them through `AtomicAmount`
(`object-model.md`) — ASCII digits only, no sign/decimal/exponent/underscore,
no leading zero, must fit u128. A JSON *number* is never a valid amount.
Floats are rejected everywhere in the money path.
